The building blocks of Solana fees

Compute units, base fees, and priority fees

Every Solana transaction consumes "compute units" — a measure of CPU work. The base fee is fixed and trivially cheap. The priority fee is optional but determines your position in the validator's execution queue.

Priority fee formula

──────────────────────────────────────────────────────────── Priority Fee Calculation ──────────────────────────────────────────────────────────── Total Priority Fee (lamports) = (cu_price_microlamports × cu_limit) ÷ 1,000,000 Example A — casual swap (low congestion) cu_price = 1,000 microlamports/CU cu_limit = 200,000 CUs priority = (1,000 × 200,000) ÷ 1,000,000 = 200 lamports (~$0.00003) Example B — arb bot during peak contention cu_price = 500,000 microlamports/CU cu_limit = 350,000 CUs priority = (500,000 × 350,000) ÷ 1,000,000 = 175,000,000 lamports = 0.175 SOL (~$26.25) Total Cost = base_fee + priority_fee + jito_tip (if using Jito) Example B total: 5,000 + 175,000,000 + 3,000,000 (tip) ≈ 0.178 SOL ─── Key insight: set cu_limit accurately or you overpay ────── Overpaying CU limit: cu_limit=1,400,000 when you use 200,000 → pays 7× the priority fee for no benefit Best practice: simulate first, set limit = simulated_usage × 1.1

Setting compute budget in TypeScript

compute-budget.ts — the two required instructions for fee control
import { ComputeBudgetProgram, Connection, Transaction } from '@solana/web3.js'; // ── Step 1: Simulate to get actual CU usage ── async function estimateCUs(conn: Connection, tx: Transaction): Promise<number> { const sim = await conn.simulateTransaction(tx, { sigVerify: false, replaceRecentBlockhash: true, // use current blockhash for simulation commitment: 'processed', }); if (sim.value.err) throw new Error(`Simulation failed: ${JSON.stringify(sim.value.err)}`); return sim.value.unitsConsumed ?? 200_000; } // ── Step 2: Build optimal compute budget instructions ── async function buildWithBudget( conn: Connection, baseTx: Transaction, cuPriceMicro:number // microlamports per CU (your priority bid) ) { const cuUsed = await estimateCUs(conn, baseTx); const cuLimit = Math.ceil(cuUsed * 1.1); // 10% headroom // MUST come before all other instructions in the transaction const limitIx = ComputeBudgetProgram.setComputeUnitLimit({ units: cuLimit }); const priceIx = ComputeBudgetProgram.setComputeUnitPrice({ microLamports: cuPriceMicro }); const totalPriorityFee = (cuPriceMicro * cuLimit) / 1_000_000; console.log(`CUs: ${cuUsed} → limit: ${cuLimit}`); console.log(`Priority fee: ${totalPriorityFee} lamports (${(totalPriorityFee/1e9).toFixed(7)} SOL)`); return [limitIx, priceIx, ...baseTx.instructions]; } // ── Step 3: Dynamic fee estimation from recent blocks ── async function getRecentPriorityFee( conn: Connection, accounts: PublicKey[] // the hot accounts your tx touches ): Promise<number> { // Returns recent per-CU prices for these specific accounts const fees = await conn.getRecentPrioritizationFees({ lockedWritableAccounts: accounts }); // Use the 75th percentile of recent winning fees const sorted = fees.map(f => f.prioritizationFee).sort((a, b) => a - b); const p75 = sorted[Math.floor(sorted.length * 0.75)] ?? 1000; console.log(`P75 recent fee for these accounts: ${p75} microlamports/CU`); return p75; }

SIMD-0096 — the defining feature of Solana's fee model

Local fee markets vs Ethereum's global gas auction

Ethereum has one global gas price — when NFTs are minting, gas spikes for everyone including people just sending ETH. Solana's local fee markets mean fee competition is scoped to the specific accounts your transaction touches. Congestion on Orca doesn't affect your SOL transfer.

Scheduler-level (SIMD-0096)

How the scheduler creates per-account fee queues

The Solana scheduler maintains a queue for every write-locked account. Transactions are sorted by priority fee within each queue. High-fee txs touching the same account jump the queue; txs touching different accounts run in parallel and don't interfere.

Write-lock contention

When two transactions both need to write to the same account, they must execute sequentially. The higher-fee transaction goes first. This is where priority fees actually matter — only when there's write-lock contention on a specific account.

Tx A: writes to [USDC_POOL, USER_ATA] → Queue USDC_POOL
Tx B: writes to [USDC_POOL, VAULT] → Queue USDC_POOL
Tx C: writes to [SOL_STAKING_POOL] → Queue SOL_STAKING

Tx A and B compete (same queue).
Tx C runs independently in parallel.
Higher fee between A/B wins position.

Practical implication for searchers

If you're arbing a newly-launched token with low competition, 1,000 microlamports/CU wins inclusion easily. If you're competing on the SOL/USDC Orca pool during a major market move, 500,000+ microlamports/CU might be needed. Always call getRecentPrioritizationFees() with your specific accounts to gauge the current market.

Read-only accounts don't compete

Only write-locked accounts create fee queues. If your tx reads a price oracle but doesn't write to it, it doesn't compete with other txs that also read the oracle. This is why read-heavy analytics transactions are cheap even during peak load.

The MEV stack Jito built on top of Solana

Block Engine, ShredStream, bundles, and tip distribution

Jito fills the gap that Solana's lack of a public mempool creates. By running a modified validator client, Jito creates a private "express lane" where searchers submit ordered bundles and validators earn tips for including them. About 70% of Solana stake now runs Jito validators.

Jito's four components

Block Engine
Aggregates searcher bundles. Runs a tip auction. Forwards winning bundles to the current slot leader.
Relayer
Forwards normal (non-bundle) transactions to the Block Engine before the validator's TPU. Enables transaction peeking.
ShredStream
Provides low-latency raw shred data from the current leader. Searchers see blocks being built in real time — before full confirmation.
Tip Distribution
On-chain program. Tips flow to validators (4%) and the staking pool (96%). Redistributed to stakers each epoch.

Bundle anatomy — what makes Jito different from just high priority fees

What a bundle guarantees

A Jito bundle is 1–5 transactions that are executed:

✓ Consecutively — no other tx executes between them
✓ Same slot — guaranteed to land together or not at all
✓ Ordered — tx[0] executes before tx[1], always
✗ Not atomic on failure — if tx[1] fails, tx[0] already ran
✗ Tip paid on failure — tip account tx is separate, always pays

Tip distribution flow

Searcher tip → Jito tip account
→ Jito TipDistribution program
→ 96% staking pool (validators + stakers)
→ 4% Jito Labs

Validators get more rewards by running Jito.
Stakers earn extra yield beyond normal inflation.

What's possible without a public mempool

MEV strategies unique to Solana's architecture

Solana's lack of a persistent public mempool eliminates classic frontrunning — you can't see pending transactions and insert yours ahead. But MEV still exists in different forms, shaped by Jito's ability to create temporary ordering guarantees.

Backrunning — dominant strategy

React to on-chain state changes

Watch for large swaps that move a pool price. The moment the swap lands, submit an arb transaction that corrects the price imbalance. Jito bundles let you submit immediately after the swap within the same slot. This is the safest and most common Solana MEV strategy.

Slot N: Large ETH/USDC swap → Orca price moves
Slot N: Your bundle: [arb tx, tip tx] → backrun in same slot
Result: You restore the price, pocket the difference
Liquidations — time-critical

Undercollateralized position clearing

Monitor lending protocols (Marginfi, Kamino, Solend) for positions that drop below their liquidation threshold. Submit the liquidation transaction with a high priority fee. Solana's 400ms slots mean positions can be liquidated before significant bad debt builds up — a key advantage over slower chains.

MarginfiKaminoSolendMango
Sandwich attacks — limited but possible

Requires Jito ordering

Without Jito, sandwiching is nearly impossible (no mempool to see victims). With Jito bundles and ShredStream, attackers can see large pending swaps and insert: buy before, let victim buy, sell after. Jito's validator community debates this — it's currently possible but widely considered harmful.

Always set minAmountOut in swaps — slippage tolerance is your only defense against sandwiching on Solana.
JIT (Just-In-Time) liquidity

Concentrated liquidity timing

Advanced: detect a large incoming swap, provide concentrated liquidity (via Orca Whirlpools) at exactly the price tick the swap will cross, earn the trading fee, then remove liquidity immediately after. Used by sophisticated actors on Uniswap v3 and now Orca.

New pool sniping

First-mover on new token launches

Monitor Raydium/Orca for new pool creation events (account creation with known program discriminator). Be the first buyer of a new token listing. High risk — many new pools are rug pulls — but significant edge if using on-chain launch detection.

Cross-market arbitrage

Long-tail and exotic pairs

Most bots watch major pairs. Edge is in newly-launched tokens, cross-DEX price gaps in obscure tokens, and cross-chain price divergences (Solana vs EVM via Wormhole-bridged assets). Less competition, but smaller absolute dollar values.

The mempool — the defining structural difference

Ethereum — the dark forest

Public mempool creates frontrunning

Every pending Ethereum transaction sits in the public mempool — visible to anyone — for potentially many seconds before inclusion. Flashbots was created specifically to solve this "dark forest" problem where generalized frontrunners copy-paste profitable transactions and submit them with higher gas.

// Classic Ethereum frontrunning (simplified)
User sends: swap(USDC→ETH, minOut=95)
Bot sees it in mempool
Bot sends: same swap, gasPrice += 1 gwei
Bot's tx mines FIRST
Price moves against user
User gets less ETH than expected
Solution: Flashbots private relay — txs go directly to builders, bypassing the public mempool entirely.
Solana — no persistent mempool

Transactions go directly to the leader

Solana transactions are sent via UDP directly to the current leader's Transaction Processing Unit (TPU). There's no shared waiting room. By the time most searchers could see a pending transaction, it's already been processed — or expired.

// Solana tx flow (no mempool)
User sends: swap(USDC→SOL, minOut=95)
→ Goes directly to leader's TPU
→ Processes in current/next slot
→ No waiting room for bots to see it

Exception: Jito Relayer routes txs
through Block Engine first (~10ms peek)
Jito Relayer creates a brief window where the block engine can see incoming txs — the only frontrunning surface on Solana.

The most harmful MEV — how it differs between chains

Sandwich attacks: Ethereum vs Solana

Sandwich attacks directly harm ordinary users. On Ethereum they're rampant (public mempool makes victims easy to find). On Solana they're rare but not impossible — Jito infrastructure enables them for well-resourced actors. Understanding both helps you protect your users.

How a sandwich attack works

Victim's tx detected
Attacker sees victim swapping 100 ETH → USDC with 1% slippage tolerance. Price will move.
Frontrun (buy before)
Attacker buys USDC before victim. Price of USDC rises. Attacker's tx has higher gas/priority.
Victim executes
Victim gets worse rate — price already moved by attacker's purchase. Slippage tolerance is their only protection.
Backrun (sell after)
Attacker sells USDC immediately after victim at the now-higher price. Profit extracted from victim's slippage.

Why Solana is much harder to sandwich

Technical barriers on Solana

✓ No public mempool — can't see victim transactions in advance
✓ 400ms slots — tiny window even if you do see a tx
✓ Parallelism — harder to guarantee ordering without Jito
✓ Small slippage on tight pools limits maximum extraction
✗ Jito Relayer creates a ~10ms window of tx visibility
✗ Jito bundles CAN order 3 txs around a victim's tx
✗ ShredStream provides real-time block construction data

How to protect your users on both chains

Set tight slippage tolerance

Use 0.1–0.5% for stable pairs. Sandwich profit is bounded by slippage tolerance.

Use Flashbots Protect (Ethereum)

Route transactions through private RPC so they never appear in the public mempool.

Always pass minAmountOut (Solana)

The on-chain slippage check is your only protection. Never pass minAmountOut=0.

Break large swaps into smaller ones

A single massive swap is an obvious sandwich target. Split across multiple transactions or use TWAP execution.

The future: PBS on Solana?

SIMD-0228 and beyond

Solana's roadmap includes formal Proposer-Builder Separation at the protocol level (similar to Ethereum's PBS post-Merge). This would create a dedicated builder market on Solana, potentially reducing the outsized advantage that co-located validators have today. Until then, Jito remains the de facto MEV infrastructure — both enabling legitimate searcher activity and creating the sandwich attack surface the community debates.
