Decision guide

Choose Solidity when…

You're deploying to Ethereum or any EVM chain. Your priority is the largest dev ecosystem, maximum tooling, and DeFi composability with existing Uniswap/Aave/Compound contracts.

Choose Rust when…

You're building on Solana, NEAR, or Polkadot. You need extreme throughput, low latency, or are writing chain infrastructure / validators. You want compile-time memory safety guarantees.

Solidity — data types and storage layout

Types, storage slots, and memory

Every state variable in a Solidity contract occupies 32-byte "slots" on the EVM. Understanding this layout is critical for gas optimization and avoiding storage collision bugs in upgradeable contracts.

Storage slot packing — gas critical

// SPDX-License-Identifier: MIT // pragma solidity ^0.8.20; contract StorageLayout { // Each slot = 32 bytes. Pack small types together to save gas. // ❌ INEFFICIENT: three separate 32-byte slots (96 bytes total) bool public flagA; // slot 0 (wastes 31 bytes) uint256 public amount; // slot 1 bool public flagB; // slot 2 (wastes 31 bytes) // ✅ EFFICIENT: packs into two slots (64 bytes total) uint128 public valueA; // slot 3, bytes 0–15 uint64 public valueB; // slot 3, bytes 16–23 ← packed! uint32 public valueC; // slot 3, bytes 24–27 ← packed! bool public flagC; // slot 3, bytes 28 ← packed! uint256 public bigVal; // slot 4 (needs full slot) // Mappings always occupy their own slot (content stored at keccak256(key, slot)) mapping(address => uint256) public balances; // slot 5 mapping(address => mapping(address => uint256)) public allowances; // slot 6 // Struct — fields packed in declaration order struct Proposal { address proposer; // 20 bytes uint64 deadline; // 8 bytes ← fits in same slot as address! uint32 voteCount; // 4 bytes ← still in same slot bool executed; // 1 byte ← still in same slot (32 bytes total) string description; // new slot (dynamic) } Proposal[] public proposals; // dynamic array — length at slot 7, elements at keccak(7) }

Memory, storage, and calldata — the three data locations

storage

On-chain, persistent

State variables live here. Costs 20,000 gas to write a new slot (SSTORE). Most expensive. Persists between transactions.

uint256 public counter; // storage counter++; // 5,000 gas (update)
memory

Ephemeral, per-call

Temporary variables inside a function. Erased after the call. Much cheaper than storage. Required keyword for arrays and strings in functions.

function foo(string memory name) { uint256[] memory arr = new uint256[](5); }
calldata

Read-only input data

Function arguments from external callers. Cheapest — never copied. Use for external functions receiving arrays or strings. Cannot be modified.

function bar( uint256[] calldata ids // cheapest! ) external { ... }

Solidity — functions, events, and OOP

Functions, modifiers, inheritance

Solidity combines familiar OOP patterns with blockchain-specific constructs — events for cheap logging, modifiers for reusable guards, and interfaces for protocol composability.

Function anatomy

contract FunctionGuide { uint256 private _balance; address public owner; // ── Custom errors (EIP-838) — 4x cheaper than require("string") ── error Unauthorized(address caller); error InsufficientBalance(uint256 requested, uint256 available); // ── Events — stored in tx receipt logs (not state), very cheap ── event Deposited(address indexed sender, uint256 amount); // indexed = filterable event Withdrawn(address indexed to, uint256 amount); // ── Modifier — reusable guard, inserted at the _; position ── modifier onlyOwner() { if (msg.sender != owner) revert Unauthorized(msg.sender); _; // function body executes here } // pure: reads no state, writes no state — cheapest to call function mul(uint256 a, uint256 b) public pure returns (uint256) { return a * b; } // view: reads state, writes nothing — free to call externally function getBalance() external view returns (uint256) { return _balance; } // payable: function can receive ETH (msg.value > 0) function deposit() external payable { require(msg.value > 0, "No ETH sent"); _balance += msg.value; emit Deposited(msg.sender, msg.value); } // State-changing + access controlled — costs gas to call function withdraw(uint256 amount) external onlyOwner { if (amount > _balance) revert InsufficientBalance(amount, _balance); _balance -= amount; (bool ok,) = msg.sender.call{value: amount}(""); require(ok, "Transfer failed"); emit Withdrawn(msg.sender, amount); } }

Interfaces, inheritance, and libraries

// Interface: defines the ABI contract any implementation must satisfy interface IERC20 { function transfer(address to, uint256 amount) external returns (bool); function balanceOf(address account) external view returns (uint256); event Transfer(address indexed from, address indexed to, uint256 value); } // Abstract base contract with shared logic abstract contract Ownable { address public owner; error NotOwner(); constructor() { owner = msg.sender; } modifier onlyOwner() { if(msg.sender!=owner) revert NotOwner(); _; } } // Library: stateless reusable code, called with delegatecall or linked library MathLib { function sqrt(uint256 y) internal pure returns (uint256 z) { if (y > 3) { z = y; uint256 x = y / 2 + 1; while (x < z) { z = x; x = (y / x + x) / 2; } } else if (y != 0) { z = 1; } } } // Concrete implementation inheriting from two bases contract MyToken is IERC20, Ownable { using MathLib for uint256; // attach library to uint256 type mapping(address => uint256) private _bal; function transfer(address to, uint256 amount) external override returns (bool) { _bal[msg.sender] -= amount; _bal[to] += amount; emit Transfer(msg.sender, to, amount); return true; } function balanceOf(address a) external view override returns (uint256) { return _bal[a]; } function sqrtOfBalance(address a) external view returns (uint256) { return _bal[a].sqrt(); // library called on uint256 value } }

Solidity — security patterns and common exploits

Security — the most critical skill

Smart contracts hold real value and are immutable once deployed. One bug can drain millions. These are the most important attack vectors every Solidity developer must understand and prevent.

1. Reentrancy — the $60M DAO hack pattern

Vulnerable — state updated after external call
contract Vulnerable { mapping(address=>uint) public bal; function withdraw() external { uint amt = bal[msg.sender]; // ❌ External call BEFORE state update (bool ok,) = msg.sender.call{ value: amt }(""); require(ok); // ❌ Attacker re-enters above line // before this runs: bal[msg.sender] = 0; } }
Secure — Checks-Effects-Interactions (CEI)
contract Secure { mapping(address=>uint) public bal; bool private _locked; modifier noReentry() { require(!_locked, "reentrant"); _locked = true; _; _locked = false; } function withdraw() external noReentry { uint amt = bal[msg.sender]; // 1. CHECK bal[msg.sender] = 0; // 2. EFFECT (bool ok,) = msg.sender // 3. INTERACT .call{value:amt}(""); require(ok); } }

2. tx.origin vs msg.sender — phishing attack vector

Never use tx.origin for auth
contract PhishableWallet { address public owner; function transfer(address to, uint amt) external { // ❌ tx.origin = original EOA signer // Attacker contract tricks owner into // calling it — tx.origin is still owner! require(tx.origin == owner); payable(to).transfer(amt); } }
Always use msg.sender
contract SafeWallet { address public owner; function transfer(address to, uint amt) external { // ✅ msg.sender = direct caller // Attacker contract's address, not owner's // So require() fails for attacker require(msg.sender == owner, "not owner"); payable(to).transfer(amt); } }

3. Other critical vulnerability checklist

Integer overflow (pre-0.8)

Now auto-checked in 0.8+

Before Solidity 0.8, uint8(255) + 1 == 0. Post-0.8 it reverts. For older contracts, use OpenZeppelin SafeMath. For unchecked math gas optimization, use unchecked { } block consciously.

Oracle manipulation

Never use spot price as oracle

Reading pool.getReserves() inline is manipulable via flash loans in the same transaction. Use Chainlink price feeds or TWAP (time-weighted average price) from Uniswap v3 for any price-sensitive logic.

Access control

Role-based access matters

Use OpenZeppelin Ownable for single-admin, AccessControl for multi-role. Never leave initialize() unguarded on upgradeable contracts — set roles in the constructor or with initializer protection.

Front-running (MEV)

Tx ordering is public

Mempool transactions are visible before inclusion. Commit-reveal schemes protect sensitive actions. Slippage parameters in DEX trades protect against sandwich attacks. Consider using private RPC endpoints (Flashbots Protect) for sensitive txs.

Rust — core language concepts for blockchain devs

Ownership, errors, and traits

Rust's ownership system is its defining feature — a compile-time memory model that eliminates entire classes of bugs with zero runtime overhead. Understanding it is the key to writing safe, fast Solana programs.

Ownership and borrowing

// RULE 1: each value has exactly one owner at a time let s1 = String::from("hello"); // s1 owns the String let s2 = s1; // ownership MOVES to s2 // println!("{}", s1); // ❌ compile error: s1 is moved (dropped) println!("{}", s2); // ✅ s2 is the owner now // RULE 2: borrowing — reference without taking ownership (&T) fn get_len(s: &String) -> usize { s.len() } // borrows, doesn't own let s3 = String::from("world"); let len = get_len(&s3); // pass reference: s3 still valid after call // RULE 3: mutable borrow — only ONE &mut ref at a time (prevents data races) let mut s4 = String::from("hello"); let r1 = &mut s4; // let r2 = &mut s4; // ❌ compile error: two mutable borrows r1.push_str(" world"); // ✅ only r1 holds the mutable borrow // Clone: deep copy when you actually need two owners (expensive) let s5 = String::from("data"); let s6 = s5.clone(); // explicit heap copy — both s5 and s6 valid // Copy types (stack-only): integers, bool, char — always copied, not moved let x: u64 = 42; let y = x; // x is still valid — integers implement Copy trait

Result and error handling — no exceptions in Rust

use std::num::ParseIntError; // Result<T, E> — either Ok(value) or Err(reason). Must be handled explicitly. fn parse_amount(s: &str) -> Result<u64, ParseIntError> { s.trim().parse::<u64>() // returns Result — propagate or handle } // ? operator: returns Err early if the value is Err (like try/catch but at compile time) fn process(input: &str) -> Result<u64, ParseIntError> { let amount = parse_amount(input)?; // if Err, returns early with Err Ok(amount * 2) } // Pattern matching — exhaustive, compiler enforces all cases are handled match process("100") { Ok(v) => println!("Got: {}", v), // ✅ success case Err(e) => println!("Error: {}", e), // ❌ error case — must handle! } // Custom error enum — best practice for blockchain programs use thiserror::Error; #[derive(Debug, Error)] pub enum VaultError { #[error("insufficient balance: need {needed}, have {available}")] InsufficientBalance { needed: u64, available: u64 }, #[error("unauthorized signer")] Unauthorized, #[error("amount must be greater than zero")] ZeroAmount, }

Traits — Rust's interface system

// Trait = interface contract. Structs implement traits. pub trait Validate { fn validate(&self) -> Result<(), VaultError>; fn description(&self) -> &str { "generic item" } // default impl } pub struct Transfer { pub amount: u64, pub recipient: String } impl Validate for Transfer { fn validate(&self) -> Result<(), VaultError> { if self.amount == 0 { return Err(VaultError::ZeroAmount); } Ok(()) } fn description(&self) -> &str { "token transfer" } } // Trait bound: accept any type that implements Validate fn execute<T: Validate>(item: &T) -> Result<(), VaultError> { item.validate()?; println!("Executing: {}", item.description()); Ok(()) } // Common derived traits — added automatically via #[derive] #[derive(Debug, Clone, PartialEq, Eq, Hash)] pub struct Pubkey([u8; 32]); // Debug=printable, Clone=copyable, PartialEq=comparable

Rust — Solana and the Anchor framework

Solana programs with Anchor

Solana's account model is fundamentally different from the EVM. Programs are stateless — all data lives in separate "accounts" that programs read and write. Anchor generates the boilerplate and validates accounts at the instruction boundary.

Program Derived Addresses (PDAs)

// PDAs: deterministic addresses with no private key — owned by programs // Derived from: seeds + program_id → find_program_address → (pubkey, bump) // The bump ensures the resulting point is off the ed25519 curve (no private key) // Client-side (TypeScript) — derive the PDA const [vaultPDA, bump] = PublicKey.findProgramAddressSync( [Buffer.from("vault"), owner.publicKey.toBuffer()], program.programId ); // On-chain (Rust/Anchor) — verify PDA in account constraints #[derive(Accounts)] pub struct Initialize<'info> { #[account( init, // create this account payer = owner, // owner pays rent-exemption space = 8 + Vault::INIT_SPACE, // discriminator + struct size seeds = [b"vault", owner.key().as_ref()], // PDA derivation seeds bump // stores canonical bump )] pub vault: Account<'info, Vault>, #[account(mut)] pub owner: Signer<'info>, // must sign the transaction pub system_program: Program<'info, System>, }

A complete Anchor program

use anchor_lang::prelude::*; declare_id!("Vault111111111111111111111111111111111111111"); #[program] pub mod token_vault { use super::*; pub fn initialize(ctx: Context<Initialize>) -> Result<()> { let vault = &mut ctx.accounts.vault; vault.owner = ctx.accounts.owner.key(); vault.balance = 0; vault.bump = ctx.bumps.vault; // store bump for future PDA signing Ok(()) } pub fn deposit(ctx: Context<Deposit>, amount: u64) -> Result<()> { require!(amount > 0, VaultError::ZeroAmount); // CPI: cross-program invocation — call the System program to transfer SOL let cpi_ctx = CpiContext::new( ctx.accounts.system_program.to_account_info(), system_program::Transfer { from: ctx.accounts.owner.to_account_info(), to: ctx.accounts.vault.to_account_info(), }, ); system_program::transfer(cpi_ctx, amount)?; ctx.accounts.vault.balance += amount; emit!(DepositEvent { owner: ctx.accounts.owner.key(), amount }); Ok(()) } pub fn withdraw(ctx: Context<Withdraw>, amount: u64) -> Result<()> { let vault = &mut ctx.accounts.vault; require!(amount <= vault.balance, VaultError::InsufficientBalance); // PDA-signed CPI: vault PDA signs as the source let seeds = &[b"vault", vault.owner.as_ref(), &[vault.bump]]; let signer = &[&seeds[..]]; let cpi_ctx = CpiContext::new_with_signer( ctx.accounts.system_program.to_account_info(), system_program::Transfer { from: ctx.accounts.vault.to_account_info(), to: ctx.accounts.owner.to_account_info(), }, signer, ); system_program::transfer(cpi_ctx, amount)?; vault.balance -= amount; Ok(()) } } #[account] #[derive(InitSpace)] pub struct Vault { pub owner: Pubkey, // 32 bytes pub balance: u64, // 8 bytes pub bump: u8, // 1 byte } #[event] pub struct DepositEvent { pub owner: Pubkey, pub amount: u64 } #[error_code] pub enum VaultError { #[msg("Amount must be greater than zero")] ZeroAmount, #[msg("Insufficient vault balance")] InsufficientBalance, }

Ultimate use cases — what to build with each

What to build and where

The right language depends on your target chain, performance needs, and the existing ecosystem you want to compose with. Here's the definitive breakdown by project type.

DeFi protocols

Solidity on EVMRust on Solana

AMM/DEXs: Uniswap (Solidity) or Orca/Raydium (Rust). Lending: Aave (Solidity) or Kamino (Rust). Derivatives: dYdX v3 (Solidity) or dYdX v4 (Rust/Cosmos). Stablecoins: MakerDAO (Solidity). For maximum composability with existing DeFi money legos → Solidity. For sub-second settlement and sub-cent fees → Rust/Solana.

NFT contracts and marketplaces

Primarily SolidityMetaplex on Solana

EVM: ERC-721 / ERC-1155 contracts with OpenZeppelin. Royalties via ERC-2981. Marketplace logic in separate contracts. Solana: Metaplex Token Metadata program (Rust) handles NFT standard. Compressed NFTs (cNFTs) via Merkle tree — mint 1M NFTs for $5 using Metaplex Bubblegum. Massive cost advantage for high-volume collections.

Blockchain gaming and DePIN

Rust strongly preferred

Games need high throughput and near-zero fees per transaction. Solana's parallel execution (Sealevel) handles thousands of game-state updates per second. DePIN (Decentralized Physical Infrastructure) projects like Helium and Hivemapper run on Solana for the same reasons. Use ERC-1155 on EVM only for slow-paced gaming or collectibles.

DAOs and governance

Solidity dominant

OpenZeppelin Governor (Solidity) is the standard for on-chain governance. Compound Governor Bravo, Uniswap DAO, and most major DAOs run on Ethereum. Solana-based DAOs use Realms (Rust) with SPL Governance. For maximum composability with existing DeFi treasury assets → Solidity/Ethereum.

Cross-chain bridges and oracles

SolidityRust

Bridges need contracts on every chain they connect. LayerZero endpoints are Solidity (EVM) + Rust (Solana). Wormhole uses Rust for its guardian network and has adapters for both. Chainlink oracle nodes use off-chain Rust/Go code and on-chain Solidity aggregator contracts. Both languages required for full-stack cross-chain work.

Chain infrastructure and runtimes

Rust exclusively

Solana's validator software, Substrate (Polkadot), Lighthouse (Ethereum consensus client), Reth (Ethereum execution client), and most L2 sequencer code are all written in Rust. If you want to build or customize a blockchain itself — consensus, p2p networking, state machine — Rust is the uncontested choice.
