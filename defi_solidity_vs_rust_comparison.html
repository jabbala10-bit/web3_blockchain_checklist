Side-by-side comparison of DeFi development in Solidity on Ethereum versus Rust on Solana — covering architecture, token model, AMM swap logic, flash loans, cost model, and security differences.

The fundamental split

Stateful contracts vs stateless programs

This is the root of every other difference. On Ethereum, a contract is both the logic and the storage. On Solana, the program is pure logic — all state lives in separate accounts that must be explicitly passed in on every call.

What an AMM pool looks like structurally

Ethereum — pool is one self-contained contract
// The contract IS the pool. // It holds both the logic AND the state. contract UniswapV2Pair { // State lives in storage slots address public token0; address public token1; uint112 private reserve0; // slot 2 uint112 private reserve1; // slot 2 (packed) uint256 public totalSupply; // LP tokens minted mapping(address=>uint256) public balanceOf; // LP balances // Logic lives in the same contract function swap( uint256 amount0Out, uint256 amount1Out, address to ) external { // reads reserve0, reserve1 from THIS contract's storage // calls token0.transfer(), token1.transfer() // writes updated reserves back to THIS contract's storage } } // To call it: just one address needed pair.swap(amountOut, 0, recipient);
Solana — program is stateless, accounts hold everything
// The PROGRAM is pure logic — no storage of its own. // State is in accounts passed in with every instruction. #[program] pub mod amm { pub fn swap(ctx: Context<Swap>, amount_in: u64) -> Result<()> { // ALL state arrives via ctx.accounts — nothing stored in program let pool = &ctx.accounts.pool; // pool config account let reserve_a = ctx.accounts.vault_a.amount; // SPL Token vault let reserve_b = ctx.accounts.vault_b.amount; // SPL Token vault // compute, validate, CPI to SPL Token program... Ok(()) } } // Pool state stored in a PDA data account #[account] pub struct PoolState { pub token_mint_a: Pubkey, // which token pub token_mint_b: Pubkey, pub lp_mint: Pubkey, // LP token mint pub fee_bps: u16, // fee in basis points pub bump: u8, } // Actual reserves: held in SPL Token vault accounts (vault_a, vault_b) // LP balances: held in per-user SPL Token accounts (not a mapping) // To call it: must list ALL accounts in the instruction // [pool, vault_a, vault_b, user_token_a, user_token_b, lp_mint, ...]

ERC-20 vs SPL Token — a core DeFi difference

How tokens move in each ecosystem

On Ethereum, every token is its own smart contract. On Solana, a single SPL Token program manages all tokens — each user needs a separate "associated token account" for every mint they hold. This changes how approvals, transfers, and DeFi integrations work.

The approve/transferFrom pattern vs direct signing

ERC-20 — two transactions to swap
// Step 1 (separate tx): user approves the DEX to spend tokens IERC20(tokenA).approve(dexAddress, amount); // Step 2 (separate tx): DEX pulls tokens from user function swap(address tokenIn, uint256 amountIn) external { // Pool calls transferFrom — user approved this IERC20(tokenIn).transferFrom( msg.sender, // from: user address(this), // to: this pool contract amountIn ); // ... compute output, send back ... } // Common UX pattern: approve(MAX_UINT256) once, // then call swap() many times without re-approving. // Security risk: unlimited approval to a potentially buggy contract.
SPL Token — user signs the instruction directly
// ONE transaction — no prior approval needed. // User signs the instruction; their signature IS the authority. #[derive(Accounts)] pub struct Swap<'info> { #[account(mut)] pub user: Signer<'info>, // signs the tx — IS the authority #[account(mut, token::authority = user)] pub user_token_a: Account<'info, TokenAccount>, #[account(mut, token::authority = user)] pub user_token_b: Account<'info, TokenAccount>, #[account(mut)] pub vault_a: Account<'info, TokenAccount>, // pool's reserve #[account(mut)] pub vault_b: Account<'info, TokenAccount>, // pool's reserve pub token_program: Program<'info, Token>, /* also: pool PDA, pool authority PDA, lp_mint... */ } // Client must construct and pass ALL these accounts. // Missing any → transaction fails immediately. // Downside: each new token type needs a new ATA created // (~0.002 SOL one-time cost, front-end usually creates it)

Same math, completely different plumbing

Implementing a constant-product AMM swap

The x·y=k invariant is the same on both chains. What differs is how reserves are read, how tokens move, and how the pool "signs" the transfer out — PDAs on Solana play the role that the contract address plays on Ethereum.

Solidity — Uniswap v2-style constant-product swap
// SPDX-License-Identifier: MIT // pragma solidity ^0.8.20; function swap( uint256 amountIn, bool zeroForOne, address recipient ) external nonReentrant { (uint112 r0, uint112 r1,) = getReserves(); (uint112 rIn, uint112 rOut, address tIn, address tOut) = zeroForOne ? (r0, r1, token0, token1) : (r1, r0, token1, token0); // x*y=k: amountOut = rOut*amountIn / (rIn+amountIn) // With 0.3% fee: amountIn *= 997/1000 uint256 amountInWithFee = amountIn * 997; uint256 amountOut = (uint256(rOut) * amountInWithFee) / (uint256(rIn) * 1000 + amountInWithFee); require(amountOut > 0, "Insufficient output"); // Pull tokenIn from caller (requires prior approve) IERC20(tIn).transferFrom(msg.sender, address(this), amountIn); // Push tokenOut to recipient — pool contract IS the authority IERC20(tOut).transfer(recipient, amountOut); // Update reserve state (pool's own storage slots) _update( IERC20(token0).balanceOf(address(this)), IERC20(token1).balanceOf(address(this)) ); emit Swap(msg.sender, amountIn, amountOut, recipient); }
Rust/Anchor — Orca Whirlpool-style swap
pub fn swap( ctx: Context<Swap>, amount_in: u64, minimum_out: u64, // slippage guard a_to_b: bool, ) -> Result<()> { let pool = &ctx.accounts.pool; let vault_a = &ctx.accounts.vault_a; let vault_b = &ctx.accounts.vault_b; // reserves read from SPL Token vault accounts let (r_in, r_out) = if a_to_b { (vault_a.amount, vault_b.amount) } else { (vault_b.amount, vault_a.amount) }; // same x*y=k math, but use u128 to avoid u64 overflow let amount_in_fee = (amount_in as u128) * 997; let amount_out = ((r_out as u128) * amount_in_fee) / ((r_in as u128) * 1000 + amount_in_fee); let amount_out = amount_out as u64; require!(amount_out >= minimum_out, AmmError::SlippageExceeded); // Transfer IN: user signs — no prior approve needed token::transfer(CpiContext::new( ctx.accounts.token_program.to_account_info(), Transfer { from: ctx.accounts.user_token_in.to_account_info(), to: ctx.accounts.vault_in.to_account_info(), authority: ctx.accounts.user.to_account_info(), // user signs } ), amount_in)?; // Transfer OUT: pool PDA signs (no private key — program signs) let seeds = &[b"pool", pool.token_mint_a.as_ref(), pool.token_mint_b.as_ref(), &[pool.bump]]; token::transfer(CpiContext::new_with_signer( ctx.accounts.token_program.to_account_info(), Transfer { from: ctx.accounts.vault_out.to_account_info(), to: ctx.accounts.user_token_out.to_account_info(), authority: ctx.accounts.pool_authority.to_account_info(), // PDA }, &[seeds] // PDA signs on behalf of the pool ), amount_out)?; emit!(SwapEvent { amount_in, amount_out, user: ctx.accounts.user.key() }); Ok(()) }
Key math difference: Solidity uses uint256 natively and can handle large intermediate values. In Rust/Solana, token amounts are u64 (max ~18.4 billion with 9 decimals) — always cast to u128 for intermediate calculations to prevent overflow.

Flash loans — same concept, completely different implementation

Atomic borrowing in Solidity vs Rust

Both chains support flash loans — uncollateralized loans that must be repaid within a single atomic transaction. On Ethereum, they use a callback pattern. On Solana, they use instruction ordering within one transaction — there's no callback; the repayment instruction is simply appended after the usage instruction.

Solidity (Aave v3) — callback pattern
// ── LENDER SIDE (Aave pool contract) ── function flashLoan( address receiver, address[] calldata assets, uint256[] calldata amounts, bytes calldata params ) external { // 1. Transfer funds to receiver for (uint i = 0; i < assets.length; i++) { IERC20(assets[i]).transfer(receiver, amounts[i]); } // 2. CALLBACK: receiver must implement this interface IFlashLoanReceiver(receiver).executeOperation( assets, amounts, premiums, msg.sender, params ); // 3. Check repayment — revert entire tx if not repaid for (uint i = 0; i < assets.length; i++) { uint256 owed = amounts[i] + premiums[i]; IERC20(assets[i]).transferFrom(receiver, address(this), owed); } } // ── BORROWER SIDE ── contract Arbitrageur is IFlashLoanReceiver { function executeOperation( address[] calldata assets, uint256[] calldata amounts, uint256[] calldata premiums, address, bytes calldata ) external override returns (bool) { // 💰 Use the borrowed funds here // e.g. buy on Uniswap, sell on Curve _arbitrage(assets[0], amounts[0]); // Approve repayment (amount + fee) uint256 owed = amounts[0] + premiums[0]; IERC20(assets[0]).approve(AAVE_POOL, owed); return true; } }
Rust/Solana — instruction ordering pattern
// No callback. The client builds one transaction with // multiple instructions. Order enforces the loan lifecycle. // ── INSTRUCTION 1: borrow (transfers funds out) ── pub fn flash_borrow(ctx: Context<FlashBorrow>, amount: u64) -> Result<()> { let state = &mut ctx.accounts.flash_state; // Record snapshot BEFORE the loan state.pre_vault_balance = ctx.accounts.vault.amount; state.borrow_amount = amount; state.loan_fee = amount / 1000; // 0.1% fee // Transfer out to borrower — PDA-signed token::transfer(CpiContext::new_with_signer( ctx.accounts.token_program.to_account_info(), Transfer { from: ctx.accounts.vault.to_account_info(), to: ctx.accounts.borrower_ata.to_account_info(), authority: ctx.accounts.vault_authority.to_account_info() }, &[&[b"vault_auth", &[ctx.accounts.pool.bump]]] ), amount)?; Ok(()) // flash_state records the debt } // ── INSTRUCTION 2: borrower does whatever they want ── // (separate program instructions, e.g. arbitrage, liquidation) // All in the same atomic transaction. // ── INSTRUCTION 3: repay (must appear in same tx) ── pub fn flash_repay(ctx: Context<FlashRepay>) -> Result<()> { let state = &ctx.accounts.flash_state; let expected = state.borrow_amount + state.loan_fee; // Check vault received expected repayment let received = ctx.accounts.vault.amount - state.pre_vault_balance; require!(received >= expected, FlashError::Underpaid); Ok(()) // If this instruction is missing or fails → entire tx reverts } // ── CLIENT (TypeScript) builds the transaction ── // const tx = new Transaction() // .add(flashBorrowIx) // ix 1 // .add(arbitrageIx) // ix 2 — use the funds // .add(flashRepayIx) // ix 3 — must repay or all fails
On Solana, because accounts must be listed upfront and transactions run atomically, a flash loan protocol can verify repayment without any callback — it simply inspects account balances at instruction 3. The entire transaction reverts if any instruction fails.

Fee models and performance characteristics

Economics of DeFi on each chain

Cost and performance differences are not just quantitative — they change what DeFi products are even viable. A liquidation bot, high-frequency trading strategy, or microtransaction product that's economically impossible on Ethereum can be profitable on Solana.

What the cost difference enables

On-chain order books

Economically impossible on Ethereum (each order = gas). Serum (now OpenBook) on Solana runs a full central limit order book on-chain at $0.00001/order. Drift Protocol runs perps with an on-chain DLOB.

High-frequency liquidations

Liquidation bots on Ethereum compete on gas price (MEV). On Solana they compete on priority fees. Sub-second finality means positions can be liquidated before significant undercollateralization builds up.

DeFi composability advantage

Ethereum wins for composability — Uniswap + Aave + Curve all in one tx is trivially cheap to compose. Solana's account listing requirement means complex multi-protocol calls need careful client-side orchestration.

Solana rent-exemption — the hidden cost

Accounts cost SOL to create and hold

Every Solana account must maintain a minimum SOL balance proportional to its data size (~0.002 SOL for a basic token account, ~0.009 SOL for a Metaplex NFT metadata account). This is returned when the account is closed — it's not a fee, it's a deposit. But it means every new DeFi position, token account, or LP stake creates a real on-chain cost for users. Good DeFi protocols close accounts when done and refund rent to users.

Different chains, different attack surfaces

Security vulnerabilities unique to each

The threats are not the same. Reentrancy is Ethereum's signature vulnerability. Solana's account model creates an entirely different class of bugs — missing signer checks and account substitution attacks — that Solidity developers must learn from scratch.

Solidity — top DeFi-specific vulnerabilities

❌ Price oracle manipulation via flash loan
contract VulnerableLending { // ❌ NEVER use spot reserves as a price oracle function getPrice(address token) public view returns (uint256) { (uint112 r0, uint112 r1,) = IUniswapPair(pair).getReserves(); return (uint256(r0) * 1e18) / uint256(r1); // ❌ manipulable! } function borrow(uint256 amount) external { uint256 price = getPrice(collateral); // ❌ flash loan pumps this // Attacker flash-loans, skews pool, inflates collateral price, // borrows against it, repays flash loan. Net: drain protocol. } } // ✅ Use time-weighted average price (TWAP) instead: function getTWAP() public view returns (uint256) { // Uniswap v3 TWAP — averaged over N seconds, resistant to flash loans uint32[] memory periods = new uint32[](2); periods[0] = 1800; periods[1] = 0; // 30-minute window (int56[] memory ticks,) = IUniswapV3Pool(pool).observe(periods); return OracleLib.getQuoteAtTick(ticks, baseAmount, token, quote); }
✅ Use Chainlink for reliable price data
import {AggregatorV3Interface} from "@chainlink/contracts"; contract SecureLending { AggregatorV3Interface immutable priceFeed; constructor(address _feed) { priceFeed = AggregatorV3Interface(_feed); } function getPrice() public view returns (int256) { ( uint80 roundId, int256 price, , uint256 updatedAt, uint80 answeredInRound ) = priceFeed.latestRoundData(); // ✅ Validate the answer is fresh (not stale) require(updatedAt > block.timestamp - 3600, "Stale price"); require(answeredInRound >= roundId, "Incomplete round"); require(price > 0, "Invalid price"); return price; // manipulation-resistant aggregated price } }

Security checklist: what auditors look for on each chain

Solidity audit checklist

✗ Reentrancy via external call before state update
✗ Spot price oracle usage (flash-loanable)
✗ tx.origin used for auth instead of msg.sender
✗ Unprotected initialize() on upgradeable contracts
✗ Storage collision in proxy pattern
✗ Unlimited ERC-20 approvals to risky contracts
✗ Precision loss: division before multiplication
✗ Front-running on DEX/auction without slippage param
✓ Overflow auto-checked in Solidity 0.8+
✓ Use nonReentrant + CEI pattern religiously

Rust/Solana audit checklist

✗ Missing signer check (anyone can call protected instruction)
✗ Missing owner check (wrong program owns the account)
✗ Account substitution (fake account passed where real expected)
✗ Using unchecked AccountInfo instead of typed Account
✗ u64 arithmetic overflow in token math (use u128 for intermediate)
✗ PDA bump manipulation (always store canonical bump)
✗ Missing has_one / constraint linking related accounts
✗ CPI to unverified program (fake token program attack)
✓ No reentrancy — Solana disallows recursive CPI by default
✓ Anchor handles 80% of these checks via macros
